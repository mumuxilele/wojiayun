# V11.0 迭代总结

## 迭代背景

本轮迭代从**产品经理深度评审**视角出发，基于 V2.0 至 V10.0 十个版本的积累，继续以"客户真实使用场景"为标准进行全面体检，重点关注：

- 三端功能对称性（用户能做的，员工端必须有对应管理操作）
- 代码健壮性（重复路由、重复定义等架构隐患）
- 运营指标完整性（退款率、评价统计、库存预警等）
- 用户端内容发现路径（门店、商品等入口是否完整）
- 自动化测试覆盖率（新功能必须有对应结构验证用例）

## 本轮完成的迭代内容

### [P0] Bug 修复

#### 1. 修复 userH5 重复路由注册
- **问题**：`business-userH5/app.py` 第 2327 行存在空的 `@app.route('/')` 定义（无函数体，中间有 8 个空行），第 2336 行才是真正的实现
- **影响**：Flask 路由注册时产生重复注册警告，在某些 Flask 版本下会导致实际路由被覆盖
- **修复**：移除空的第一个定义，保留带有 `index()` 函数体的完整定义

#### 2. 扩展员工端后端 API（+7 个核心路由）

| 路由 | 方法 | 功能 |
|------|------|------|
| `/api/staff/orders/<id>/ship` | POST | 员工发货（保存物流公司+运单号，发送通知） |
| `/api/staff/orders/<id>/refund` | POST | 退款审核（approve/reject，自动回滚库存） |
| `/api/staff/orders/<id>/complete` | POST | 确认订单完成 |
| `/api/staff/members` | GET | 会员查询（支持关键词搜索+分页） |
| `/api/staff/checkin/stats` | GET | 签到统计（今日/本周/本月人数 + Top5 活跃用户） |
| `/api/staff/orders/pending-refunds` | GET | 待处理退款订单列表 |
| `/api/staff/products` | GET | 商品库存查看列表 |

所有新路由均包含：
- `@require_staff` 认证
- `ec_id / project_id` 多租户数据隔离
- `@rate_limiter.rate_limit()` 限流保护
- 关键操作触发 `notification.send_notification()` 用户通知

## 关键技术要点

1. **员工端数据隔离**：所有新接口均通过 `ec_id` / `project_id` 过滤，防止跨租户数据访问
2. **退款事务完整性**：退款拒绝时在同一事务内完成状态回滚 + 库存恢复，避免半成功问题
3. **限流保护**：发货接口 60秒/20次，退款审核 60秒/20次，防止员工误操作或接口滥用
4. **用户通知**：发货、退款批准/拒绝、订单完成均触发 `notification.send_notification()`
5. **幂等迁移**：所有迁移操作均先检查是否已存在再执行，支持安全重试
