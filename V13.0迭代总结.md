### 1.1 支付金额服务端校验 ✅ 已完成

**问题描述**:  
原 `create_payment` 接口直接使用客户端传入的 `amount` 参数，存在金额篡改风险。攻击者可构造请求将实际支付金额改为 0.01 元购买高价值商品。

**修复方案**:
- 在创建支付前，从服务端重新计算订单/预约的实际应付金额
- 与客户端传入的金额进行校验（允许 0.01 元浮点数精度误差）
- 使用服务端计算的 `actual_amount` 调用支付服务

**修复文件**: `business-userH5/app.py` (create_payment 函数)

**代码示例**:
```python
# P13新增：服务端金额校验，防止客户端篡改
if order_type == 'order':
    order = db.get_one(
        "SELECT actual_amount, order_status FROM business_orders WHERE id=%s AND user_id=%s AND ec_id=%s",
        [order_id, user['user_id'], user.get('ec_id')]
    )
    if not order:
        return jsonify({'success': False, 'msg': '订单不存在'})
    if order['order_status'] == 'paid':
        return jsonify({'success': False, 'msg': '订单已支付'})
    actual_amount = float(order['actual_amount'] or 0)
    
# 金额校验
if abs(client_amount - actual_amount) > 0.01:
    logging.warning(f"支付金额校验失败: client={client_amount}, actual={actual_amount}")
    return jsonify({'success': False, 'msg': '金额校验失败，请刷新页面重试'})

# 使用服务端金额
payment_service.payment.create_payment(..., amount=actual_amount, ...)
```

### 1.2 订单状态流转校验 ✅ 已完成

**问题描述**:  
原 `admin_update_order` 和员工端 `update_order_status` 接口可任意修改订单状态为任何值，导致订单状态机混乱。例如：已完成的订单被改为"待支付"。

**修复方案**:
1. 新增 `business-common/order_status.py` 状态流转模块
2. 定义完整的状态流转规则（7种状态 + 终态保护）
3. 在所有状态变更接口添加流转校验

**状态流转规则**:
```
pending → paid（支付后）
pending → cancelled（用户取消）
paid → shipped（发货后）
paid → refunding（申请退款）
shipped → completed（确认收货）
shipped → refunding（申请退款）
refunding → refunded（退款通过）
refunding → paid（退款拒绝，恢复支付状态）

终态（不可流转）: cancelled, refunded, completed
```

**修复文件**:
- `business-common/order_status.py` (新增)
- `business-common/__init__.py` (导出模块)
- `business-admin/app.py` (admin_update_order)
- `business-staffH5/app.py` (update_order_status)

## 二、P1 性能优化

### 2.1 库存预扣减事务优化 ✅ 已完成

**问题描述**:  
原 `create_order` 函数库存校验在事务外部，存在 TOCTOU (Time-of-Check to Time-of-Use) 竞态条件，高并发下可能导致超卖。

**修复方案**:
1. 将库存校验移入事务内部
2. 使用 `SELECT ... FOR UPDATE` 批量锁定商品记录
3. 使用带条件的 UPDATE 语句 `WHERE stock >= quantity` 双重保险

**优化要点**:
- 按商品 ID 排序后锁定，防止死锁
- 事务内先锁定再校验再扣减
- UPDATE 带条件检查，affected == 0 时回滚

### 2.2 用户统计接口 N+1 查询优化 ✅ 已完成

**问题描述**:  
原 `get_user_stats` 接口执行 7 次独立数据库查询，在高并发场景下造成数据库连接压力。

**修复方案**:  
将 7 次独立查询合并为单次查询，使用子查询方式获取所有统计：

```python
stats = db.get_one("""
    SELECT 
        (SELECT COUNT(*) FROM business_applications WHERE user_id=%s AND status='pending') AS pending,
        (SELECT COUNT(*) FROM business_applications WHERE user_id=%s AND status='processing') AS processing,
        (SELECT COUNT(*) FROM business_applications WHERE user_id=%s AND status='completed') AS completed,
        (SELECT COUNT(*) FROM business_orders WHERE user_id=%s AND deleted=0) AS orders,
        (SELECT COUNT(*) FROM business_venue_bookings WHERE user_id=%s AND deleted=0) AS bookings,
        (SELECT COUNT(*) FROM business_notifications WHERE user_id=%s AND is_read=0) AS unread_notifications,
        (SELECT COUNT(*) FROM business_user_coupons uc WHERE uc.user_id=%s AND uc.status='unused') AS unused_coupons
""", [uid]*7)
```

**优化效果**: 数据库查询从 7 次减少为 1 次
