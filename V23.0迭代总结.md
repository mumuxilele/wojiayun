**项目**: 我家云 - 社区商业服务系统
**版本**: V23.0

**主题**: 产品深度评审 + 积分商城 + 内容审核 + 架构优化

## 一、迭代背景

基于AI产品经理视角的深度评审，从V1.0到V22.0已迭代22个版本，系统功能趋于完善。
本次聚焦于：**补全核心业务闭环、提升产品安全性、建立统一架构规范**。

### 评审发现的核心问题

| 问题 | 等级 | 影响 |
|------|------|------|
| 评价内容无审核 | P1 | 合规风险，用户可能发布违规内容 |
| 积分只有获取没有消耗闭环 | P1 | 积分形同虚设，用户无消耗动力 |
| 退款不回收积分 | P1 | 用户退款后仍保留积分，经济漏洞 |
| 无统一错误码 | P2 | 前端无法统一处理错误，用户体验差 |
| 支付系统仍为Mock | P0 | 无法正式上线（保留为V24重点） |

## 二、核心变更

### 2.1 F1: 评价内容审核集成 (P1)

**问题**: V22.0已编写 `content_moderation.py` 模块，但 `create_review` 接口未调用审核，评价直接入库。

**修复方案**: 在用户端评价提交接口中集成自动审核流程。

**修改文件**: `business-userH5/app.py`

**审核逻辑**:
- risk_level 0-1: 自动通过 (approved)
- risk_level 2: 标记待人工审核 (pending_review)
- risk_level 3+: 直接拒绝，提示用户修改

**追评也集成审核**: `append_review` 接口同步增加内容审核。

**新增数据库字段** (business_reviews表):
- `moderation_status` VARCHAR(20) - 审核状态
- `moderation_result` TEXT - 审核结果JSON
- `moderated_at` DATETIME - 审核时间
- `append_moderation_status` VARCHAR(20) - 追评审核状态
- `append_moderation_result` TEXT - 追评审核结果JSON

### 2.2 F2: 积分商城系统 (P1)

**问题**: 积分系统有获取(签到/消费/兑换优惠券)但没有真正消耗闭环，用户积分积累后无处使用。

**解决方案**: 新建完整的积分商城模块。

**新增文件**: `business-common/points_mall.py`

**核心功能**:

1. **积分商品管理** (管理端)
   - 创建/编辑/上下架积分商品
   - 设置积分价格、库存、每人限兑数量
   - 商品分类管理

2. **积分兑换** (用户端)
   - 用户浏览积分商品列表
   - 积分兑换商品（事务保证：扣积分+扣库存+创建兑换单+记录积分日志）
   - 兑换订单管理（待发货/已发货/已完成）
   - 确认收货

3. **兑换发货** (员工端)
   - 员工查看待发货兑换订单
   - 填写物流单号/物流公司
   - 发货后自动通知用户

**新增API端点** (userH5):
```
GET  /api/points-mall/products              # 积分商品列表
GET  /api/points-mall/products/<goods_id>   # 商品详情
POST /api/user/points-mall/exchange          # 积分兑换
GET  /api/user/points-mall/exchanges         # 我的兑换记录
POST /api/user/points-mall/exchanges/<id>/confirm  # 确认收货
```

**新增API端点** (staffH5):
```
GET  /api/staff/points-exchanges            # 兑换订单列表
POST /api/staff/points-exchanges/<id>/ship   # 兑换商品发货
```

**新增数据库表**:
- `business_points_goods` - 积分商品表
- `business_points_exchanges` - 积分兑换记录表

### 2.3 F3: 退款积分回收 (P1)

**问题**: 用户申请退款成功后，订单消费产生的积分未回收，造成经济漏洞。

**修复方案**: 在员工端退款批准逻辑中增加积分自动回收。

**修改文件**: `business-staffH5/app.py`

**回收逻辑**:
1. 查询该订单产生的积分收入（从 `business_points_log` 中查找 `ref_id=order_id` 且 `log_type='earn'` 的记录）
2. 扣减用户积分（保证不低于0）
3. 记录积分回收日志（`log_type='refund_deduct'`，`ref_type='refund'`）

### 2.4 F4: 统一错误码体系 (P2)

**问题**: 各接口错误返回格式不统一，前端难以统一处理。

**解决方案**: 建立标准错误码体系和统一响应构建器。

**新增文件**: `business-common/response_builder.py`

**错误码分段**:
| 段 | 范围 | 说明 |
|----|------|------|
| 10xxx | 10000-10999 | 通用错误（参数、认证、权限、限流） |
| 11xxx | 11000-11999 | 用户端业务错误（订单、支付、购物车、评价） |
| 12xxx | 12000-12999 | 积分与会员相关错误 |
| 13xxx | 13000-13999 | 营销活动相关错误 |
| 20xxx | 20000-20999 | 员工端业务错误 |
| 30xxx | 30000-30999 | 管理端业务错误 |

**统一响应格式**:
```json
{
  "success": true/false,
  "code": 0/错误码,
  "msg": "提示信息",
  "data": {}
}
```

**便捷方法**:
- `ResponseBuilder.success(data, msg)` - 成功响应
- `ResponseBuilder.error(error_code, msg)` - 错误响应
- `ResponseBuilder.paginated(items, total, page, page_size)` - 分页响应
- `success_resp()` / `error_resp()` / `paginated_resp()` - 快捷函数

## 六、系统架构优化

### 修改前
```
评价 → 直接入库 → 展示（无审核）
积分 → 获取(签到/消费) → 兑换优惠券 → 无处消耗
退款 → 批准 → 不回收积分
错误 → 各接口自定义格式，无统一标准
```

### 修改后
```
评价 → 自动审核(0-1通过/2待审/3拒绝) → 展示
积分 → 获取(签到/消费) → 兑换商品/兑换优惠券 → 闭环
退款 → 批准 → 自动回收该订单产生的积分
错误 → 统一错误码体系(10xxx-30xxx) → 标准响应格式
```
