## 一、产品评审总结

### 1.1 当前系统现状

经过V1.0-V15.0的迭代，系统已具备以下核心能力：

| 能力域 | 已实现功能 | 完善度 |
|--------|-----------|--------|
| 用户端H5 | 申请单、订单、场地预约、商品购买、门店浏览、签到、积分、优惠券、消息通知、地址管理、评价 | 75% |
| 员工端H5 | 申请处理、订单管理、预约管理、会员管理、退款处理、统计报表、反馈管理 | 70% |
| 管理端PC | 场地管理、申请审核、订单管理、门店管理、会员管理、数据统计、优惠券管理 | 65% |
| 基础服务 | 认证鉴权、数据隔离、审计日志、限流、XSS防护、缓存、文件上传、支付框架、消息通知、WebSocket框架 | 85% |
| 测试 | 58个结构验证测试用例 | 60% |

### 1.2 产品问题矩阵（深度评审）

#### 🔴 P0 - 业务场景缺失（影响核心收入）

| 问题 | 现状 | 影响 | 建议 |
|------|------|------|------|
| 真实支付未对接 | 只有模拟支付和框架 | 无法进行真实交易 | 对接微信/支付宝沙箱 |
| 会员等级体系缺失 | 单一会员身份 | 无差异化服务，黏性低 | 实现消费升级机制 |
| 营销工具缺失 | 无促销活动 | 转化率低，难以促活 | 满减/折扣/秒杀/新人礼 |

#### 🟠 P1 - 功能深度不足（影响用户体验）

| 问题 | 现状 | 影响 | 建议 |
|------|------|------|------|
| 积分通胀风险 | 积分无过期机制 | 积分价值稀释 | 积分过期+动态兑换 |
| 权限管理粗放 | 员工/管理员二分 | 无法精细管控 | 完整RBAC权限系统 |
| API文档缺失 | 无完整接口文档 | 接入成本高 | Swagger/OpenAPI |
| WebSocket闲置 | 有框架未应用 | 消息不及时 | 订单状态推送 |

#### 🟡 P2 - 体验优化点

| 问题 | 现状 | 影响 | 建议 |
|------|------|------|------|
| 管理端XSS未全 | 部分页面未覆盖 | 安全风险 | 全面覆盖escapeHtml |
| 数据大屏缺失 | 只有统计数字 | 决策效率低 | ECharts可视化大屏 |
| 端到端测试缺失 | 只有结构验证 | 回归风险 | Playwright集成测试 |
| 短信通知缺失 | 只有站内消息 | 触达率低 | 集成短信服务 |

## 三、详细设计方案

### 3.1 支付系统完善（V16-1/2）

**目标**：对接真实支付渠道，支持沙箱测试

**技术方案**：
```
┌─────────────────────────────────────────┐
│           PaymentService                │
├─────────────────────────────────────────┤
│  + create_payment()    ← 统一下单       │
│  + verify_callback()  ← 回调验签        │
│  + query_payment()   ← 状态查询        │
│  + refund()          ← 退款接口         │
└─────────────────────────────────────────┘
          ↓                    ↓
   微信统一下单API      支付宝即时到账API
```

**实现要点**：
- 环境变量配置：`WECHAT_APP_ID`, `WECHAT_MCH_ID`, `WECHAT_API_KEY`, `ALIPAY_APP_ID`等
- 沙箱测试：使用微信/支付宝沙箱环境
- 回调验签：使用RSA/AES验签
- 幂等设计：防止回调重复处理

### 3.2 会员等级体系（V16-3/4）

**目标**：基于消费金额/积分自动升级，提供差异化服务

**等级设计**：
| 等级 | 名称 | 升级条件（累计消费） | 权益 |
|------|------|---------------------|------|
| L1 | 普通会员 | 0元 | 基础权益 |
| L2 | 铜牌会员 | 500元 | 9.5折 + 积分1.2倍 |
| L3 | 银牌会员 | 2000元 | 9折 + 积分1.5倍 + 优先预约 |
| L4 | 金牌会员 | 5000元 | 8.5折 + 积分2倍 + 专属客服 |
| L5 | 钻石会员 | 10000元 | 8折 + 积分3倍 + 免费场地 |

**数据库设计**：
```sql
-- 会员等级表
CREATE TABLE business_member_levels (
    id INT PRIMARY KEY AUTO_INCREMENT,
    level_code VARCHAR(20) NOT NULL UNIQUE,
    level_name VARCHAR(50) NOT NULL,
    min_amount DECIMAL(10,2) DEFAULT 0,
    discount_rate DECIMAL(5,2) DEFAULT 1.00,
    points_rate DECIMAL(5,2) DEFAULT 1.00,
    privileges TEXT,
    sort_order INT DEFAULT 0,
    status VARCHAR(20) DEFAULT 'active'
);

-- 会员等级记录
ALTER TABLE business_members ADD COLUMN member_level VARCHAR(20);
ALTER TABLE business_members ADD COLUMN total_consume DECIMAL(12,2) DEFAULT 0;
```

### 3.3 营销工具（V16-5/6/7）

#### 3.3.1 满减优惠活动

**活动规则**：
- 门槛：满X元减Y元
- 叠加：可与会员折扣叠加
- 库存：每人限领/全场总量

**数据库设计**：
```sql
CREATE TABLE business_promotions (
    id INT PRIMARY KEY AUTO_INCREMENT,
    promo_name VARCHAR(100) NOT NULL,
    promo_type VARCHAR(20) NOT NULL, -- 'full_reduce'/'discount'/'seckill'
    reduce_amount DECIMAL(10,2),    -- 满减金额
    min_amount DECIMAL(10,2),       -- 最低消费
    discount_rate DECIMAL(5,2),     -- 折扣率
    start_time DATETIME NOT NULL,
    end_time DATETIME NOT NULL,
    total_stock INT DEFAULT 0,
    per_limit INT DEFAULT 1,
    status VARCHAR(20) DEFAULT 'pending',
    ec_id VARCHAR(50),
    project_id VARCHAR(50)
);
```

#### 3.3.2 限时秒杀

**核心逻辑**：
- 定时开放：秒杀场次（10:00/14:00/20:00）
- 库存扣减：预占库存 -> 支付锁定 -> 超时释放
- 限购：每人每场限N件
- 价格保护：秒杀价不与会员折扣叠加

### 3.4 积分过期机制（V16-8）

**设计规则**：
- 积分有效期：12个月（从获得日开始计算）
- 过期提醒：到期前30天/7天/1天推送通知
- 过期执行：每日凌晨定时任务检查并扣除过期积分

**实现逻辑**：
```python
# 定时任务：每日执行
def expire_points_task():
    # 查找12个月前获得的积分记录
    expired_records = db.query("""
        SELECT user_id, SUM(points) as total 
        FROM business_points_log 
        WHERE created_at < DATE_SUB(NOW(), INTERVAL 12 MONTH)
        AND expired = 0
        GROUP BY user_id
    """)
    
    for record in expired_records:
        # 扣除积分
        deduct_points(record['user_id'], record['total'], 'expired')
        # 发送通知
        notification.send(..., '您的{amount}积分已过期')
```
