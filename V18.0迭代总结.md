## 评审发现问题

本轮迭代基于对 193 个 API 路由、47 个 HTML 页面、46 个公共模块的深度产品评审，发现以下关键问题：

| 优先级 | 问题描述 | 影响范围 |
|--------|---------|---------|
| P0 | 员工端商品管理仅有只读接口，无法调整库存/上下架 | 日常运营无法执行 |
| P0 | 支付确认后无后处理：积分未发放、会员等级未升级 | 核心商业逻辑断层 |

| P2 | 用户端首页缺少促销/地址快捷入口 | 功能可发现性差 |

## 本次变更内容

### P0 修复：员工端商品管理完善

**文件**: `business-staffH5/app.py`

新增 3 个接口：

| 接口 | 方法 | 功能 |
|------|------|------|
| `/api/staff/products` | GET（升级） | 支持 status 筛选（all/active/inactive/low_stock/out_of_stock），返回低库存/缺货数量统计 |
| `/api/staff/products/<id>/stock` | POST | 库存调整（add/subtract/set 三种模式），写入审计日志 |
| `/api/staff/products/<id>/status` | POST | 上架/下架切换，写入审计日志 |

代码特性：
- 所有接口均有 `@require_staff` 认证保护
- 库存调整有类型校验（仅 add/subtract/set）
- 数量不合法时返回 400 错误
- 每次操作通过 `audit_log.log_action()` 留存记录

---

### P0 修复：支付系统闭环

**文件**: `business-common/payment_service.py`

新增 `_post_payment_hooks()` 函数，在事务提交后异步触发：

```
支付确认
    └── conn.commit()
    └── _post_payment_hooks()
            ├── 1. 站内通知（send_notification）
            ├── 2. WebSocket 实时推送（push_notification）
            ├── 3. 积分发放（按会员等级倍率：1元×倍率=积分）
            │       └── 写入 business_points_log
            └── 4. 更新累计消费 → 检查等级升级（MemberLevelService.update_total_consume）
```

关键设计：
- 每个子功能独立 try-except，任一失败不影响其他
- 仅 `order_type == 'order'` 发放积分（预约支付不重复给分）
- 等级升级通过 `MemberLevelService.update_total_consume()` 自动计算

### P2 优化：购物车结算流程升级

**文件**: `business-userH5/cart.html`

旧流程：`下单 → alert提示 → 跳转订单`  
新流程：`下单 → 创建支付单 → 跳转payment.html → 确认支付 → 支付成功页`

- 下单中禁用按钮防重复提交
- 支付单创建失败时降级回旧流程
- 传递 pay_no、amount、order_no 到支付页
