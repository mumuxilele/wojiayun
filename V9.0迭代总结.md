### 2.2 订单取消未回滚库存 [已修复]

**问题描述**:  
用户取消pending订单时，已扣减的商品库存未能回滚，导致库存不匹配。

**修复方案**:  
在 `business-userH5/app.py` 的 `cancel_order` 接口中，取消订单时自动查询订单明细，逐一回滚商品库存和销售数。

**代码示例**:
```python
# 取消订单时回滚库存
items = db.get_all(
    "SELECT product_id, quantity FROM business_order_items WHERE order_id=%s", [order_id]
)
for item in items:
    db.execute(
        "UPDATE business_products SET stock=stock+%s, sales_count=GREATEST(0, sales_count-%s)",
        [item['quantity'], item['quantity'], item['product_id']]
    )
```

**影响范围**:  
- 订单取消流程完整性提升
- 库存准确性有保障

### 2.3 下单流程缺少收货地址 [已修复]

**问题描述**:  
虽然V8.0添加了用户地址簿功能，但下单流程未集成地址选择，物流无法正确投递。

**修复方案**:  
在 `create_order` 接口中新增 `address_id` 参数，下单时将用户选择的地址快照保存到 `business_orders.shipping_address` 字段。

**数据库变更**:  
- 新增 `business_orders.shipping_address` 字段（VARCHAR 500）

**前端集成**:  
下单时从用户地址簿中选择默认地址或手动选择。

## 三、P1 功能增强

### 3.1 订单完整生命周期管理

#### 3.1.1 订单退款流程 [新增]

**用户端API**:
```
POST /api/user/orders/<order_id>/refund
请求: { reason: "退款原因" }
响应: { success, order_id, order_no }
```

**功能说明**:
- 用户可申请退款（pending订单除外）
- 订单状态变更为 `refunding` 
- 自动回滚库存和优惠券
- 触发用户通知

**管理端API**:
```
POST /api/admin/orders/<order_id>/refund
请求: { action: "approve" | "reject", reason: "审核意见" }
```

**审核流程**:
- 同意: 订单状态变为 `refunded`，用户通知已退款
- 拒绝: 订单恢复为 `paid`，库存重新扣减，用户通知被拒

**新增数据库字段**:
```sql
ALTER TABLE business_orders ADD COLUMN refund_reason VARCHAR(500);
ALTER TABLE business_orders ADD COLUMN refunded_at DATETIME;
```

#### 3.1.2 订单发货处理 [新增]

**管理端API**:
```
POST /api/admin/orders/<order_id>/ship
请求: { tracking_no: "SF123...", logistics_company: "顺丰" }
响应: { success, msg }
```

**功能说明**:
- 订单状态: `paid` → `shipped`
- 记录物流单号和物流公司
- 自动通知用户（物流信息）

**新增数据库字段**:
```sql
ALTER TABLE business_orders ADD COLUMN tracking_no VARCHAR(100);
ALTER TABLE business_orders ADD COLUMN logistics_company VARCHAR(100);
ALTER TABLE business_orders ADD COLUMN shipped_at DATETIME;
ALTER TABLE business_orders ADD COLUMN completed_at DATETIME;
```

#### 3.1.3 用户确认收货 [新增]

**用户端API**:
```
POST /api/user/orders/<order_id>/confirm
响应: { success, msg }
```

**功能说明**:
- 仅已发货订单可操作
- 状态变更为 `completed`
- 记录完成时间

### 3.2 用户签到系统 [新增]

#### 3.2.1 签到核心功能

**用户端API**:
```
POST /api/user/checkin
响应: { success, data: { continuous_days, reward_points, bonus } }

GET /api/user/checkin/status
响应: { success, data: { checked_in, continuous_days, month_count, total_days, recent, today_reward } }

GET /api/user/checkin/rank
响应: { success, data: [ { user_name, member_level, checkin_days, max_continuous }, ... ] }
```

#### 3.2.2 签到奖励机制

| 连续天数 | 每日奖励 | 7天达成奖励 | 总积分 |
|---------|---------|-----------|-------|
| 第1天 | 1积分 | - | 1 |
| 第2天 | 2积分 | - | 2 |
| ... | ... | - | ... |
| 第7天 | 7积分 | +10 | 17 |
| 第8天 | 1积分(重置) | - | 1 |

**特性**:
- 连续签到天数自动计算
- 7天循环重置（但达成7天有额外10积分奖励）
- 月度统计、本月排行
- 防重复签到（10秒限流）

#### 3.2.3 新增数据库表

```sql
CREATE TABLE business_checkin_logs (
    id BIGINT AUTO_INCREMENT PRIMARY KEY,
    user_id VARCHAR(64) NOT NULL,
    checkin_date DATE NOT NULL,
    continuous_days INT DEFAULT 1,
    reward_points INT DEFAULT 0,
    created_at DATETIME DEFAULT CURRENT_TIMESTAMP,
    
    UNIQUE KEY uk_user_date (user_id, checkin_date),
    KEY idx_user_id (user_id),
    KEY idx_checkin_date (checkin_date)
) COMMENT='用户签到记录';
```

## 七、性能与安全

### 7.1 缓存优化

签到相关查询使用缓存：
- 签到状态缓存120秒
- 排行榜缓存60秒

### 7.2 限流保护

新增限流规则：
```python
@rate_limiter.rate_limit(limit=1, window_seconds=10)  # 签到10秒内只能1次
@rate_limiter.rate_limit(limit=5, window_seconds=60)  # 退款60秒内最多5次
@rate_limiter.rate_limit(limit=10, window_seconds=60) # 发货60秒内最多10次
```

### 7.3 数据隔离

所有新增API均遵循数据隔离规则：
- `ec_id` / `project_id` 过滤
- 防止跨项目数据泄露
